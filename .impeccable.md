# RAE Scale — Design Context

## Users

**Primary audience:** Designers and developers building type systems for web projects. They come to RAE Scale when setting up a new project's typography or refining an existing one. They want numbers they can trust and copy directly into their codebase or Figma file.

**Secondary audience:** The broader design/dev community discovering the tool via shared URLs. They may be less familiar with modular scales and need the interface to be self-explanatory.

**Job to be done:** Calculate a harmonious type scale, verify it looks right at each size, and export production-ready code — all in under a minute.

## Brand Personality

**Three words:** Friendly, clear, practical.

**Voice:** Direct and helpful without being patronizing. The UI speaks through well-chosen defaults and obvious controls, not through explanatory text. Labels are concise. Interactions are immediate.

**Emotional goals:** Clarity and calm focus. The user should feel "everything is obvious, I can focus on my design decisions" and "no distractions, just the tool and my work."

## Aesthetic Direction

**Visual tone:** Clean, spacious, light. Typography-forward — the content (the scale itself) is the hero, not the chrome. The UI should feel like a well-designed page in a type specimen book.

**References:**
- **v0.app** — The overall structure and polish level. Single-page tool feel.
- **linear.app** — Precision UI, tight feedback loops, no unnecessary decoration.

**Anti-references:**
- No retro or skeuomorphic elements (no faux-3D, textures, nostalgia)
- Not dense or IDE-like — the interface should breathe
- No gradients, illustrations, or emoji-heavy UI
- No generic SaaS dashboard patterns (cards-with-icons landing pages)

**Theme:** Light mode only (v1). Zinc palette throughout. Color is reserved for functional meaning (strand indicators: blue/purple).

## Design Principles

1. **Content is the interface.** The type scale preview IS the product. Controls exist to serve it, not compete with it. Every pixel of chrome should justify its presence.

2. **Instant feedback, zero modes.** Changing any control immediately updates the preview and the export. No "generate" buttons, no loading states for calculations. The URL is always the source of truth.

3. **Generous space, not empty space.** Whitespace is used intentionally to create hierarchy and breathing room — 32px sidebar padding, 28px preview padding, 28px gaps between control groups. Space communicates structure.

4. **Precision through typography.** Monospace for values (px, rem, step numbers). Proportional for labels and UI text. The typography itself demonstrates competence — if a type scale tool has bad typography, nothing else matters.

5. **Progressive disclosure.** Single base by default. Dual base behind a toggle. Advanced options (prefix, rounding) lower in the sidebar. The tool is simple on first encounter and powerful on closer inspection.

## Tech Stack

- **Framework:** Next.js 15+ (App Router)
- **Components:** shadcn/ui v2 (Base UI primitives, NOT Radix)
- **Styling:** Tailwind CSS v4 with OKLch color tokens
- **UI Font:** Inter (via next/font/google)
- **Mono Font:** Geist Mono (code, values)
- **Icons:** lucide-react
- **URL State:** nuqs

## Accessibility

- WCAG AA compliance (4.5:1 contrast for text, 3:1 for UI elements)
- Keyboard navigable controls
- ARIA tab semantics on view/format toggles
- Screen reader compatible labels
- `prefers-reduced-motion` respected where animations exist
